1 · Splitting principles

Following the smart / dumb (container / presenter) pattern:

  • Presentation components never inject HttpClient, Router, ActivatedRoute, or any feature service. They take data via input() and emit events via output(). They are pure, easy to test in isolation, and reusable across screens.
  • Domain components are wired to routes. They inject services (HTTP, router, signals/state), translate route params into queries, orchestrate child presentation components, and translate child events back into service calls.
  • Layout shells (header / sidebar / bottom nav) consume a tiny amount of state (current user, active route) — kept as domain so the dumb pieces beneath stay pure.
Heuristic: If swapping the component into a Storybook story without mocking any service is awkward, it belongs in the domain tier. If you could ship the component to a design-system package without touching the rest of the app, it belongs in presentation.

5 · Services & cross-cutting providers

Injected only into domain components — never into presentation components. Each is a providedIn: 'root' singleton backed by Angular signals.

BillService

list(), getById(id), create(dto), update(id, dto), delete(id), markSettled(id). Wraps HttpClient calls to /api/bills.

PaymentService

list(), create(dto). Wraps /api/payments.

BalanceService

Derives outstanding balance and monthly stats from the bill + payment streams; exposes balance, thisMonth, received, behindByDays as signals.

ActivityService

Merges bills + payments into a single chronological feed with running-balance projection. Used by Dashboard (recent 5) and History (full timeline).

ProfileService

Holds the user + partner names and avatar initials; persisted via /api/profile. Consumed by sidebar, settings page and form previews.

DialogService

Thin wrapper over Angular CDK Dialog for the delete-bill confirmation. Opened from BillDetailPage.
